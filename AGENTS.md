# AGENTS.md

## 目的
このリポジトリは、チェロのパーツ交換（エンドピン、弦、駒、テールピース、ペグ、魂柱など）について調べた内容を、後から検証・更新できる形で蓄積します。本ファイルは、作業者（人間またはAI）が「どのように記事を追加・修正するか」を統一するための手順と基準を定めます。

## スコープ
扱う内容
- パーツ交換の候補比較、選定基準、取り付け・調整に伴う注意点を扱います。
- 実測や体感の記録と、外部情報（メーカー仕様、工房の見解、文献）を分離して記述します。
- 個人の楽器固有差が大きい領域は、一般化し過ぎない書き方を優先します。

扱わない内容
- 危険行為を助長する具体的手順（例：魂柱立て、駒削りの具体作業の断定手順）を詳細化しません。
- 医療・法務・保険判断に該当する助言を行いません。

## リポジトリ構成（前提）
- `docs/` 配下に Jekyll（Minimal Mistakes テーマ）のサイトがあります。
- GitHub Pages のソースを `docs` に設定すると公開できます。

## 記事（ページ）追加・更新ルール
### 1. 追加場所
新規記事は原則として `docs/` 配下に追加します。既存カテゴリがある場合は、それに合わせて配置します。

### 2. 記事の書式（推奨テンプレート）
記事冒頭に要点を置き、次に根拠と判断基準を置きます。不確実な点は「未検証」と明記し、断定表現を避けます。

推奨セクション構成は次の通りです。
- TL;DR（結論と前提条件を2〜4行で書きます）。
- 背景（なぜ交換・検討するのかを書きます）。
- 選定基準（重量、材質、弦高影響、音色傾向、耐久性、入手性、費用などを並べます）。
- 候補比較（比較軸を固定し、表現を揃えます）。
- 作業・調整に伴う注意（安全側に倒し、工房相談の推奨を書きます）。
- 結果・所感（主観は主観として書き、条件を添えます）。
- 参考リンク（一次情報を優先して列挙します）。

### 3. 主観と客観の分離
- 「体感」「自分の楽器では」などの主観は、必ず条件（弦、弓、部屋、録音有無）を添えて書きます。
- メーカー仕様や計測値などの客観情報は、出典（URL、書籍名）を付けて書きます。

### 4. 安全・リスク表現
- 工房領域の作業は「推奨しない」「相談する」方針で統一します。
- 断定的な作業指示を書かず、注意喚起と判断材料に留めます。

## ローカル実行（確認手順）
依存関係を入れてローカルでプレビューします。

1. Ruby環境がある前提で `bundle install` を実行します。
2. 次のコマンドでサイトを起動します。
   - `bundle exec jekyll serve --source docs --baseurl "/cello-parts-log"`

## 変更の最小化（編集方針）
- 既存記事の構造（見出し階層、用語、表現）を尊重して、差分を小さくします。
- 文章の言い換えだけのコミットを避け、必ず「意味のある改善」を含めます。
- 参考リンクの更新は、可能ならアーカイブ（Internet Archiveなど）も併記します。

## コミット・PR（推奨）
- コミットメッセージは「何をどう変えたか」が分かる短文にします（例：`Add endpin material comparison`）。
- 1PRは1テーマに寄せ、記事追加と大規模整形を混ぜません。

## エージェント（AI）向け作業制約
- 新規記事作成時は、既存記事の文体・粒度を優先して合わせます。
- 外部情報を参照した場合は、必ず一次情報リンクを優先し、推測で穴埋めしません。
- 不明点が残る場合は、未確定として残し、TODOとして箇条書きにします。
