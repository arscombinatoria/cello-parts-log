# AGENTS.md

## この文書の目的

- 本文書は、このリポジトリで作業する AI エージェントの判断基準と編集規約を定義するためのものです。
- 対象は、事実確認、記述方針、安全配慮、ページ構成、比較表の整合性、コミット/PR 時の注意点を含みます。
- 人間向けの一般的な寄稿ガイドではなく、AI が編集方針を誤らないための実務規約として扱ってください。

## リポジトリの目的とスコープ

- チェロのパーツ交換・調整に関する情報を記録し、後から検証・更新できる形で共有することを目的とします。
- 最優先は、チェリストが比較検討しやすい情報設計にすることです。
- 単なる仕様羅列ではなく、選定判断に必要な要素が揃っていることを重視します。

## コンテンツ編集の原則

- 主観と客観を分けて記述します。
- 記述では少なくとも次の観点を意識します。
  - 音色傾向
  - 反応性
  - 弾きやすさ
  - 耐久性
  - 価格帯
  - 相性条件（弦・弓・楽器傾向・演奏環境など）
- 可能であれば、初心者/上級者/楽器傾向ごとの判断材料を併記します。
- 既存の文体・粒度・章立てに合わせ、不要な全面書き換えや無関係な整形だけの変更は避けます。

## 情報源と根拠の扱い

- 事実確認は一次情報（メーカー公式、公式仕様、一次資料）を最優先します。
- 一般的な聴感や演奏者間で共有される傾向は、ラベルを明示した補助情報として扱って構いません。
- 次の情報を混同しないでください。
  - 仕様
  - 公式説明
  - 販売情報
  - 一般的な評判
  - 筆者の所感
- 主観を書く場合は、条件（弦、弓、楽器傾向、演奏環境など）を必ず添えます。
- 未検証の音質断定は避けます。
- 販売店の宣伝文は、誇張や信頼性を確認し、事実としてそのまま転記しません。
- 確証が薄い内容は断定せず、推定・一般論であることを明示します。
- 参考文献や一次情報リンクを可能な限り明記し、必要に応じてアーカイブ併記を検討します。

## 安全に関する原則

- 魂柱、駒加工など事故や損傷につながる作業は手順化しません。
- これらの作業は「専門の工房・職人に相談する」方向へ誘導する文体に統一します。
- 危険作業について、DIY を後押しする断定的な書き方は避けます。

## ページ構成と情報設計

- `/docs` 配下のページ種別（トップ、カテゴリ、詳細）を意識して編集します。
- トップページは、比較・導入・一覧を担うページとして扱います。
- 子ページがある場合は、メーカー別またはシリーズ別の詳細を担うページとして扱います。
- メーカー別ページでは会社紹介に終始せず、シリーズ/モデル差が比較できる構成を推奨します。
- 特定カテゴリ（例: 弦）で「トップで比較、子ページで詳細化」する構成は許容します。
- ただし特定ページ構造への過度な依存は避け、他パーツにも適用できる一般ルールとして整理します。

## 比較表と用語の統一

- 比較表の列名は、類似ページ間で可能な限り統一します。
- 列を追加・変更した場合は、同系統ページの表も点検し整合性を確保します。
- 列名統一の目的は、読者の横断比較を容易にすることです。

## AI エージェントの作業手順

- 編集前に、対象ページだけでなく同粒度の類似ページ・兄弟ページを確認します。
- 事実確認は一次情報を優先し、不確実な点は断定せず明示します。
- 変更は 1 つの目的に収束させ、差分の理由を説明できる状態で実施します。
- 不要な翻訳、大量生成、文体の過剰変更は避けます。

## コミット・PR・検証ルール

- 1 PR 1 トピックを原則とし、文章整形のみの変更は極力避けます。
- 変更前後で以下の検証コマンド実行を基本とします。
  - `npm run lint`
  - `npm test`
  - `npm run build`
- 大きな変更・大規模生成・安全に関わる変更では、PR 説明に理由と概要を明記します。

## 開発メモ（Docusaurus 前提）

- Node.js LTS と npm を使用します。
- 依存関係は `npm ci` で復元し、開発サーバーは `npm start` を使用します。
- Docs は `/docs` 配下に配置し、Front Matter でタイトルやスラッグを設定します。
- サイトのルート構成は `docs/index.md` を参照してください。
